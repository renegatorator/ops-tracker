# Repository overview — Ops Tracker

A high-level, code-anchored map of the Ops Tracker application. Every claim below points at a concrete file, table, or symbol so that future agents can verify behaviour without re-deriving it.

This document deliberately overlaps slightly with `README.md` and `docs/ARCHITECTURE.md`; where it does, it stays consistent and adds detail those docs do not. Where they conflict, the **code wins** and this doc is wrong.

---

## 1. App purpose

Ops Tracker is a Jira-style project / issue tracker built as a portfolio-grade demonstration of a production Next.js + Supabase stack. The marketing-style description lives in `README.md` (lines 1–11). The shipped capabilities, all observable in code, are:

- multi-project workspace with human-readable issue keys (e.g. `OPS-12`),
- Kanban board per project (drag-and-drop status transitions),
- virtualized issues list with offset pagination, sort, and filters,
- per-issue detail page with status / assignee / type / title / description editing under role-aware permissions,
- soft-delete (archive) of issues,
- admin CRUD for workflow statuses,
- admin user list and role management,
- super-admin demo data reset and project rename,
- append-only audit trail, surfaced in an admin global view and on each issue,
- transactional email on issue creation and assignment (when Resend is configured).

There is **no** product-marketing or external-facing copy in this repo beyond the landing page (`src/components/Pages/LandingPage.tsx`) and the README.

## 2. Business domain

The domain model is project-scoped issue tracking. Database tables (defined in `supabase/migrations/`):

| Table | Defined in | Notes |
|---|---|---|
| `user_profiles` | external prerequisite (see `docs/SUPABASE_MIGRATIONS.md` §1) | App-managed view of `auth.users` with `role app_role` and `full_name`. |
| `issue_statuses` | `supabase/migrations/20260403140000_phase1_issues_audit_statuses.sql` | Ordered workflow states; `slug` unique; `is_terminal` flag. |
| `issues` | same migration, extended in `20260412200000_projects_members_issues_scope.sql` and `20260413100000_issue_type_and_default_statuses.sql` | Adds `project_id`, `issue_number`, `issue_key`, `issue_type`, `deleted_at`. |
| `audit_log` | `20260403140000_phase1_issues_audit_statuses.sql` | Append-only; `actor_id`, `action`, `entity_type`, `entity_id`, `metadata jsonb`. |
| `projects` | `20260412200000_projects_members_issues_scope.sql` | `key` is uppercase slug `^[A-Z][A-Z0-9]{1,9}$`; tracks `last_issue_number` for sequence allocation. |
| `project_members` | same | Composite PK `(project_id, user_id)`, role enum `project_member_role` = `'lead' \| 'member'`. |

Domain enums:

- `app_role` (Postgres) → `'user' \| 'admin' \| 'super_admin'` — mirrored in `src/lib/auth/types.ts` (`AppRoles`, `AppRoleValues`).
- `issue_type` (Postgres) → `'bug' \| 'ticket'` — mirrored in `src/features/issues/issueTypeUtils.ts` (`IssueTypes`). UI labels these **Bug** and **Task**; never compare the raw string.
- `project_member_role` (Postgres) → `'lead' \| 'member'` — mirrored in `src/features/projects/types.ts` (`ProjectMemberRole`).

Issue identity has **two** forms used interchangeably in the UI:
- UUID `issues.id` (used for `/issues/[id]`),
- human key `issue_key = '<project.key>-<issue_number>'` generated by Postgres trigger `issues_before_insert_set_number` (defined in `20260412200000_projects_members_issues_scope.sql`, lines 219–264) and kept in sync by `issues_before_update_sync_key`.

Project membership is enforced **at write time** by `issues_before_write_check_members` (same migration, lines 266–308): both `reporter_id` and (if non-null) `assignee_id` must exist in `project_members` for the issue's project.

## 4. Frontend / backend boundaries

There is **one** Next.js process. There is no separate API service. The split is therefore between *server-only code* and *client-only code* inside the same repo:

**Server-only code** (executed in Server Components, Server Actions, or the proxy):

- `src/lib/supabase/server.ts` — `createServerClient` with cookie reader/writer (`cookies()` from `next/headers`).
- `src/lib/supabase/proxy.ts` — `refreshSupabaseSession` for the middleware proxy.
- `src/lib/auth/session.ts` — `getSession`, `getUserAuthContext`, `requireUser`, `requireRole`.
- `src/lib/audit/log-audit.ts` — has `import "server-only"` at the top.
- `src/lib/email/send-issue-*.ts` — both `import "server-only"`; instantiate `Resend` only when `RESEND_API_KEY` is set.
- `src/lib/recaptcha.ts` — `import "server-only"`; verifies tokens against `https://www.google.com/recaptcha/api/siteverify`.
- `src/lib/env.ts` — wraps `process.env` with `env(name)` plus typed flag readers (`isDemoResetEnabled`, `getResendApiKey`, …).
- All `features/*/actions.ts` (`"use server"` directive at top of each).
- All `features/*/service.ts` are imported only by server actions and Server Components, so they run server-side, but are not marked `"server-only"` — see §15.
- Server Components in `src/app/[locale]/.../page.tsx` and layouts (no `"use client"`).
- `src/features/issues/prefetch-issue-queries.ts` — `import "server-only"`.

**Client-only code** (`"use client"` at top):

- `src/lib/supabase/client.ts` — `createBrowserClient` for any browser-side Supabase usage. **Note:** in current code, all data flow goes through Server Actions; the browser client exists but is not invoked from feature hooks. Confirmable by grep: no `createClient` import from `@/lib/supabase/client` outside `client.ts` itself.
- `src/components/Providers/QueryProvider.tsx` — wraps the tree in `QueryClientProvider` with `staleTime: 60s`.
- All hooks under `src/features/*/hooks/` and most `src/features/*/components/*.tsx`.
- `src/components/Layout/WorkspaceShellClient.tsx` (the `AppShell` shell).

**Boundary contract:** The browser never talks to Supabase directly today. Client hooks call Server Actions; Server Actions go through `@/lib/supabase/server` (cookie-bound, RLS-applied). This means:

- session cookies live in HTTP-only cookies set by `@supabase/ssr`,
- the publishable key is shipped to the client (it is `NEXT_PUBLIC_*`) but never used by the client today,
- the only client-side fetches to external services are Google reCAPTCHA's grecaptcha script (loaded from `<Script>` in the login page — see CSP allowlist in `next.config.ts`).

## 5. Deployment assumptions

Confirmed in code / config:

- **Target host:** Vercel. `next.config.ts` sets a Content-Security-Policy that whitelists `https://vitals.vercel-insights.com` (line 78). `README.md` §"Deployment" calls out Vercel as the recommended host.
- **Node runtime:** `>=20.19.0` (`package.json` `engines.node`). CI uses `20.19.0` (`.github/workflows/ci.yml` line 19).
- **Build commands:** `next build` / `next start` (`package.json` scripts).
- **Sass aliasing:** `next.config.ts` registers a custom `sassOptions.importers[].findFileUrl` so `@/...` works in SCSS modules, mirroring the TypeScript path alias from `tsconfig.json` (`paths."@/*": ["./src/*"]`).
- **Security headers:** set in `next.config.ts` `headers()`. CSP includes `connect-src 'self' https://*.supabase.co wss://*.supabase.co`. The `wss://` line is forward-looking — there is no Realtime subscriber today (see §10).
- **Compression / etags:** `compress: true`, `generateEtags: false`, `poweredByHeader: false`.
- **Database:** managed Supabase. Migrations in `supabase/migrations/` are applied via `npm run db:push` (Supabase CLI) or pasted into the SQL Editor in filename order. There is no app-side migration runner.
- **Email:** Resend. Optional. When `RESEND_API_KEY` is missing, every `send*EmailIfConfigured` function is a no-op — never throws (`src/lib/email/send-issue-created-email.ts` lines 15–24).
- **CI:** `.github/workflows/ci.yml` runs `npm run lint` and `npx tsc --noEmit` on PRs into `master / main / development`. Playwright is **not** run in CI today (browser download is even disabled via `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD: "1"`).
- **Local DB scripts:** `db:start` / `db:stop` / `db:reset` in `package.json` for local Supabase via `npx supabase`. No `supabase/config.toml` is checked in (see `supabase/.gitignore`).

## 6. Routing, middleware, and locales

- One Next.js middleware: `src/proxy.ts`. Per Next 16, the file **must** be named `proxy.ts`, not `middleware.ts`. The `.cursor/rules/architecture.mdc` rule explicitly warns against creating both.
- The proxy chains two responsibilities (in this order):
  1. `intlMiddleware` from `next-intl/middleware` — matches against `routing` (`src/i18n/routing.ts`: `locales: ["en","si","de"]`, `defaultLocale: "en"`, `localePrefix: "as-needed"`).
  2. `refreshSupabaseSession` (`src/lib/supabase/proxy.ts`) — calls `supabase.auth.getUser()` so refreshed cookies land on the response.
- For `/` the proxy also detects preferred locale from `NEXT_LOCALE` cookie or `Accept-Language` (`src/proxy.ts` lines 10–38) and redirects to `/<locale>` for non-default locales.
- Matcher (`src/proxy.ts` line 58): `["/", "/((?!api|_next|_vercel|.*\\..*).*)"]`. The `/api/*` path therefore bypasses the proxy entirely.
- Navigation must use re-exports from `@/i18n/navigation` (`Link`, `redirect`, `useRouter`, `usePathname`). Direct imports from `next/link` or `next/navigation` would break locale handling. This is enforced by convention (see `.cursor/rules/architecture.mdc`) and is consistent across all routes.
- URL helpers live in `src/lib/routes.ts` (`routes.*`, `issueDetailPath`, `projectBoardPath`, …) and `src/i18n/localized-path.ts` (`localizedPath(locale, pathname)` — used when calling `revalidatePath` or generating absolute email URLs).

## 7. Data flow

Standard read flow for client-side queries:

```
React component
  → TanStack Query hook (e.g. useIssuesList)
    → Server Action (e.g. listIssues in features/issues/actions.ts)
      → getUserAuthContext() — auth check + role load
      → Zod parse via *.safeParse
      → service.ts (Supabase client built from cookies())
        → PostgREST (RLS as authenticated role)
      ← rows mapped into typed result
    ← { ok, data } | { ok: false, errorKey }
  ← throws IssuesQueryError on failure (so TanStack treats it as error)
```

Standard mutation flow:

```
Component → mutation hook (useMutation)
  → Server Action
    → auth + Zod
    → service mutation
    → logAudit({ actorId, action, entityType, entityId, metadata })
    → optional sendIssueXxxEmailIfConfigured(...)  (best-effort)
    → revalidatePath(...) and/or revalidateTag(ISSUES_CACHE_TAG)
  ← typed result
  ← onSuccess: queryClient.invalidateQueries({ queryKey: ... })
```

Initial / detail page flow uses the **prefetch + hydrate** pattern:

- `src/features/issues/prefetch-issue-queries.ts` runs server-side, populates a `QueryClient` via `prefetchQuery`, then `dehydrate(queryClient)` is wrapped by `<HydrationBoundary>`.
- Used by:
  - `src/app/[locale]/issues/[id]/page.tsx` — global issue detail by UUID.
  - `src/app/[locale]/projects/[projectKey]/issues/[issueNumber]/page.tsx` — resolves project key + issue number into a UUID first, then prefetches.
- The dashboard does **not** use this pattern; instead `src/features/dashboard/getDashboardData.ts` is called directly from `src/app/[locale]/dashboard/page.tsx` and the result is passed as a prop (Server Component → Client Component).

Cache invalidation surfaces:

- `revalidatePath(localizedPath(locale, ...), "page" | "layout")` for RSC-rendered pages (`src/features/issues/actions.ts` lines 47–67, `src/features/projects/actions.ts` lines 37–48).
- `revalidateTag(ISSUES_CACHE_TAG, "max")` from `src/features/issues/cache.ts` — currently the tag constant is defined and called on writes, but no `fetch(..., { next: { tags: [ISSUES_CACHE_TAG] } })` callsite uses it (no matches under `src/`). It is set up for future use.
- TanStack Query keys come from `features/<feature>/keys.ts` (`issueQueryKeys`, `auditQueryKeys`, `projectQueryKeys`, `userQueryKeys`).

`QueryClient` defaults: `staleTime: 60_000` ms (`src/components/Providers/QueryProvider.tsx` line 17 and the per-page hydration clients). Devtools render only when `NODE_ENV === "development"`.

## 8. Auth flow

- **Provider:** Supabase Auth (email + password) via `signInWithPassword` (`src/components/Pages/LoginPage/LoginPage.tsx` line 33). No OAuth, no magic-link in the current code.
- **reCAPTCHA v3:** the login Server Action verifies a token first via `verifyRecaptchaToken` (`src/lib/recaptcha.ts`). When `RECAPTCHA_SECRET_KEY` is unset, verification short-circuits to `true` (line 16–18). When set, score < 0.5 fails. CSP in `next.config.ts` whitelists `https://www.google.com` and `https://www.gstatic.com` for the script.
- **Cookies:** all session reads/writes go through `@supabase/ssr`. Three matched constructors:
  - server / RSC: `src/lib/supabase/server.ts` (uses `cookies()` from `next/headers`),
  - middleware: `src/lib/supabase/proxy.ts` (sets cookies on the `NextResponse`),
  - browser: `src/lib/supabase/client.ts` (currently unused at runtime — see §4).
- **Middleware refresh:** `refreshSupabaseSession` calls `supabase.auth.getUser()` on every request that the proxy matches, so token refresh and cookie rotation happen on each navigation.
- **Session check helpers** (`src/lib/auth/session.ts`):
  - `getSession()` → `{ user }` (no profile load).
  - `getUserAuthContext()` → `{ user, role, fullName }` after a `user_profiles` lookup; returns `null` if not signed in or profile missing.
  - `requireUser(locale)` redirects to `routes.login` if no user.
  - `requireRole(locale, allowed)` redirects to login (no user) or `/dashboard` (wrong role).
- **Logout:** Server Action `signOutAction(locale)` in `src/lib/auth/actions.ts` calls `supabase.auth.signOut()` and redirects to `/login`. Triggered by `LogoutForm` (button posts to a server action).
- **Redirect rules:**
  - `/` (root) — signed-in users are redirected to `/dashboard` (`src/app/[locale]/page.tsx` line 23).
  - `/dashboard`, `/issues`, `/issues/[id]`, `/projects`, `/projects/[projectKey]/board`, `/projects/[projectKey]/issues/[issueNumber]` — server pages call `getUserAuthContext()` and redirect to `/login` if null.
  - `/admin/**` — `src/app/[locale]/admin/layout.tsx` calls `requireRole(locale, AdminAccessRoles)`, which redirects regular users to `/dashboard`.
  - `/admin/settings` — additionally calls `requireRole(locale, SuperAdminRoles)` in its page (line 33).

## 9. Role system

Three roles, defined as both a Postgres enum (`app_role`) and a TS literal union (`src/lib/auth/types.ts`):

```ts
AppRoles = { USER: "user", ADMIN: "admin", SUPER_ADMIN: "super_admin" }
AdminAccessRoles = ["admin", "super_admin"]
SuperAdminRoles = ["super_admin"]
```

Helpers: `isAdminAccessRole(role)`, `isSuperAdminRole(role)`, `parseAppRole(value)`.

Two layers of enforcement are always in place; both must agree:

**Layer 1 — server actions (`src/lib/auth/rbac.ts`):**

- `assertRole(ctx, allowed)` throws `ForbiddenError` if `ctx.role` is not in `allowed`.
- Each Server Action that mutates does: `getUserAuthContext()` → unauthorized if null → optional `assertRole(ctx, AdminAccessRoles | SuperAdminRoles)` → Zod parse → service call.
- `assignIssue`, `softDeleteIssue`, `createIssueStatus` / `update*` / `delete*`, `createProject`, `updateProject`, `addProjectMember`, `removeProjectMember`, `listUserProfilesForAdmin`, `updateUserRole`, `listAuditLogsForAdmin`, `resetDemoData`, `renameProject` — all call `assertRole`. `renameProject` and `resetDemoData` use `SuperAdminRoles`.
- Read actions (`listIssues`, `getIssue`, `listIssueStatuses`, `listProjects`, `getProjectByKey`, …) only require an authenticated user; finer access is enforced by RLS.

**Layer 2 — RLS in Postgres** (defined across all migrations under `supabase/migrations/`). Critical helpers added by `20260412140000_fix_user_profiles_rls_recursion.sql` are `SECURITY DEFINER` SQL functions:

- `ops_auth_is_admin()`, `ops_auth_is_super_admin()`, `ops_auth_is_staff()` — read `user_profiles.role` for `auth.uid()` without re-triggering RLS recursion.
- `ops_auth_can_access_project(p_project_id uuid)` — staff OR row in `project_members` for that project. Defined in `20260412200000_projects_members_issues_scope.sql` (lines 191–214).

RLS summary per table:

| Table | SELECT | INSERT | UPDATE | DELETE |
|---|---|---|---|---|
| `issue_statuses` | any authenticated | staff | staff | staff |
| `issues` | non-deleted in accessible projects (any authenticated); soft-deleted only staff | reporter must equal `auth.uid()` and project must be accessible | staff (any), or `user` role if `reporter_id`/`assignee_id` matches and `deleted_at IS NULL` | staff only |
| `audit_log` | staff | actor must equal `auth.uid()` | — | — |
| `projects` | accessible projects via `ops_auth_can_access_project` | staff | staff | staff |
| `project_members` | accessible projects | staff | staff | staff |
| `user_profiles` | own row OR staff | (no insert policy in migrations — see §15) | super_admin: any; admin: only rows whose current role is `user`, and only to `user`/`admin` | (no delete policy in migrations) |

Note the **app duplicates** RLS for issue editing in `src/features/issues/permissions.ts` so the UI can hide actions when not allowed. The duplication is intentional (defense in depth and UX), but if RLS or the helper drift, only RLS still gates actual writes.

Role transitions (admin can demote/promote `user`↔`admin`; super_admin can do anything) are enforced **both** in RLS (`phase8_user_profiles_update_admin_user_rows` policy) and in `src/features/users/service.ts` (`updateUserRole`, lines 66–76). Audit row is logged on success.

## 10. Supabase usage

Client construction is centralized in `src/lib/supabase/`. The publishable key + URL are required (`src/lib/env.ts`); the server-only secret key is **not** used anywhere.

- **No service-role key** is referenced in code (grep confirms): every server-side operation runs as the authenticated user, with RLS in force.
- **`SECURITY DEFINER` functions** are the only mechanism that bypasses RLS, and they only do role / membership lookups.
- **Schemas / tables** referenced from app code: `user_profiles`, `issues`, `issue_statuses`, `audit_log`, `projects`, `project_members` (all under `public`). Plus `pg_tables` from the diagnostic API route.
- **Foreign-key embeds** (PostgREST `select`):
  - `issueSelect` in `src/features/issues/service.ts` (lines 25–45) embeds `issue_statuses`, `projects!inner`, `assignee:user_profiles!issues_assignee_id_fkey`.
  - `auditSelect` in `src/features/audit/service.ts` (lines 12–24) embeds `actor:user_profiles!audit_log_actor_id_fkey`.
  - `listProjectMembers` (`src/features/projects/service.ts` lines 146–194) embeds `user_profiles`.
- **Triggers** owned by the app (in migrations):
  - `set_updated_at` — generic `updated_at = now()` (used on `issues` and `projects`).
  - `issues_before_insert_set_number` — allocates next `issue_number` from `projects.last_issue_number` and builds `issue_key`.
  - `issues_before_write_check_members` — enforces project membership for reporter and assignee.
  - `issues_before_update_sync_key` — keeps `issue_key` consistent if `project_id`/`issue_number` change (unusual path).
- **Indexes** worth knowing about:
  - Phase 1 partial indexes on `issues (status_id|assignee_id|reporter_id|created_at)` `WHERE deleted_at IS NULL`.
  - `20260411120000_phase12_list_sort_audit_indexes.sql` adds composite list/sort indexes (`updated_at desc, id desc`, `title asc, id asc`, `status_id, created_at desc, id desc`, …) and audit indexes (`entity_type, created_at desc`, `entity_type, entity_id, created_at desc`).
  - Project-scoped: `issues_project_id_idx`, `issues_project_status_idx`, unique `issues_project_issue_number_uidx (project_id, issue_number)`.
- **Pagination:** `listIssues` (`src/features/issues/service.ts`) supports both `mode: "offset"` and `mode: "cursor"`; the cursor is just the offset wrapped in base64url JSON (`encodeListCursor` line 50). Real keyset pagination is acknowledged as future work in `docs/ARCHITECTURE.md` line 225. URL sync (`src/features/issues/list-url-params.ts`) supports **only** offset mode.
- **Search** in `listIssues` strips `% _ , ( ) /` to avoid LIKE metacharacters and SQL hints (`sanitizeSearch`, line 47); applies to `title.ilike` OR `issue_key.ilike`.
- **Soft delete** is `issues.deleted_at`. All list/detail reads filter `.is("deleted_at", null)` unless an `includeClosed` flag is passed.

## 12. Logging and audit architecture

**Audit log** is the structured, persisted record of mutations. Application logging (stdout) is sparse and unstructured.

### Persisted audit (`audit_log` table)

- Schema and RLS in `supabase/migrations/20260403140000_phase1_issues_audit_statuses.sql` (lines 57–69 schema, 266–284 policies).
- Append-only by RLS: only `INSERT` and `SELECT` are granted (line 304); `INSERT` requires `actor_id = auth.uid()`; `SELECT` requires staff.
- Insertion path: `src/lib/audit/log-audit.ts` `logAudit({ actorId, action, entityType, entityId, metadata })`. Returns `false` on failure and logs to `console.error`; **does not throw**, so a failed audit will not abort a successful mutation. The single exception is `resetDemoData` (`src/features/settings/actions.ts` lines 79–93) where a missing audit row is treated as fatal and returns `errorKey: "settings.errors.auditFailed"`.

Action keys actually emitted (verified by reading every `actions.ts`):

| `action` | Emitter | Notes / metadata |
|---|---|---|
| `issue.create` | `createIssue` | `{ title, project_id, issue_key }` |
| `issue.update` | `updateIssue` | `{ fields, issue_key }` |

Why some things live where they do:

- **`src/proxy.ts` lives at `src/`** because Next 16 Turbopack expects exactly that path, and exactly one such file. Do not introduce `src/middleware.ts`.
- **`src/app/[locale]/...` everywhere**: every user-facing route is locale-prefixed (or unprefixed for `en`); the only non-`[locale]` route is `src/app/api/test-supabase/route.ts`, which is excluded from the proxy matcher.
- **Feature folders** (`src/features/<feature>/`) intentionally contain `components/` and `hooks/` so feature code is colocated. The architecture rule (`.cursor/rules/architecture.mdc`) explicitly forbids creating a global `src/hooks/` unless something is genuinely cross-cutting.
- **`src/components/` is for cross-feature UI only** (Layout shell, providers, skeletons, theme toggle, language switcher, typography, generic page chrome, the login form). It is not a kitchen sink — most domain UI lives under `src/features/*/components/`.
- **`src/utils/`** holds pure stateless helpers; today only `seoUtils.ts`. Anything stateful or with I/O belongs in `src/lib/`.

## 14. Important architectural patterns

1. **Server-first defaults.** Pages are Server Components; the client tree only opts in via `"use client"` where state or DOM APIs are needed.
2. **Server Actions as the API surface.** All mutations and most reads called from the client go through `"use server"` functions in `features/<feature>/actions.ts`. Each follows the same skeleton: auth → role check → Zod parse → service → audit + revalidate.
3. **Result objects, not exceptions.** Service and action functions return `{ ok, data } \| { ok: false, errorKey, fieldErrors? }`. Errors thrown to TanStack Query are wrapped in `IssuesQueryError` (`src/features/issues/issues-query-error.ts`) so the `errorKey` survives.
4. **Translation-keys-as-error-codes.** Validation messages in Zod schemas are translation keys (e.g. `"validation.titleRequired"`), not English strings. Mapping happens in `src/features/issues/map-errors.ts` (Zod field errors and Postgres error codes 23503 / PGRST116 / "row-level security" → `errorKey`).
5. **Prefetch + HydrationBoundary** for SSR'd queries on detail routes (`src/features/issues/prefetch-issue-queries.ts`). Avoids a flash of loading on the client.
6. **Two layers of authorization** on every mutation: app code (`assertRole`) + RLS (in SQL). `permissions.ts` adds a *third* read-side check that mirrors the user-only RLS branch — only used to hide UI controls.
7. **Issue type abstraction** (`src/features/issues/issueTypeUtils.ts`). The DB enum is `bug \| ticket`, the UI calls them Bug / Task; `IssueTypes.BUG`, `IssueTypes.TASK`, `isIssueBug`, `isIssueTask` are the only correct way to compare.
8. **TanStack Query keys as a tree.** Each feature exports a `queryKeys` factory that produces nested arrays (`["issues", "list", locale, params]`); invalidations target the appropriate sub-tree (`issueQueryKeys.lists()` invalidates all list variants, `issueQueryKeys.detail(...)` only one). This is consistent across `issueQueryKeys`, `auditQueryKeys`, `projectQueryKeys`, `userQueryKeys`.
9. **Optimistic updates** on hot paths (e.g. `useUpdateIssue` in `src/features/issues/hooks/useUpdateIssue.ts`): `onMutate` snapshots and patches the detail cache; `onError` rolls back; `onSuccess` invalidates lists and shows a notification.
10. **Locale-aware path helpers.** Mutation actions call `revalidatePath(localizedPath(locale, route), "page" \| "layout")`; emails use `${siteUrl}${localizedPath(locale, issueDetailPath(id))}`. The `localizedPath` helper enforces the `as-needed` prefixing rule consistently.
11. **Email is best-effort.** `send*EmailIfConfigured` functions (`src/lib/email/`) early-return when `RESEND_API_KEY` or recipient is missing, log on Resend error, and always return `void` — never bubble to fail the mutation.
12. **CSP + headers tuned to actual dependencies.** `next.config.ts` allows google.com and gstatic.com only because reCAPTCHA scripts/iframes are loaded; supabase.co (HTTPS + WSS) for auth + future realtime; `vitals.vercel-insights.com` for Web Vitals.

## 15. Recurring conventions

- **Imports:** sorted by `eslint-plugin-simple-import-sort`; unused removed by `unused-imports`. Path alias is `@/...` for `src/...`.
- **Navigation imports:** always from `@/i18n/navigation`. Never from `next/link` or `next/navigation`.
- **Styling:** SCSS Modules colocated next to components (`*.module.scss`), with global tokens/mixins under `src/styles/`. Mantine v8 components are preferred over raw HTML.
- **Server-only files** start with `import "server-only"` (`src/lib/audit/log-audit.ts`, `src/lib/email/*`, `src/lib/recaptcha.ts`, `src/features/issues/prefetch-issue-queries.ts`). Service files (`src/features/*/service.ts`) **do not** carry the marker today even though they are intended to be server-only — see §16.
- **Server Actions:** every file declared with `"use server"` at the top. Each exported function takes `(locale: string, raw: unknown)` (or just `locale`) — the locale is forwarded so revalidation paths can be localized.
- **Zod schemas** live in `schemas.ts` and export both the schema and `z.infer<typeof ...>` types. Messages are translation keys.
- **Query keys** are factory objects at `keys.ts`, never inline tuples in components.
- **Result type naming:** `<Feature>ActionResult<T>` / `<Feature>ActionFailure` / `<Feature>ActionSuccess<T>`. The issues feature defines the canonical `IssuesActionResult<T>`; other features import or mirror it (audit, settings, users currently re-use `IssuesActionResult` directly).
- **Translation files** (`src/translations/{en,de,si}.json`) must stay in sync. New keys go into all three (called out by the architecture rule). The audit UI silently falls back to the raw action key when the translation is missing (`useAuditTranslations`).
- **Audit metadata** for issue-related actions always includes `issue_key` (verified across `src/features/issues/actions.ts`). Other actions include enough metadata to identify the affected entity.
- **Project keys** are uppercase in DB and code; route segments tolerate URL-encoded variants (`decodeURIComponent`) but services normalise (`key.trim().toUpperCase()` before any query).
- **Soft-delete filtering** (`is("deleted_at", null)`) is the default everywhere; `includeClosed` is the explicit opt-in.

## 16. Dangerous, tightly coupled, or fragile areas

These are not opinions — they are concrete coupling points or known-rough edges visible in code today.

1. **Cross-feature reuse of issues primitives.** `src/features/audit/actions.ts` (line 5), `src/features/audit/service.ts` (line 1), `src/features/users/actions.ts` (line 6), and `src/features/settings/actions.ts` (line 7) all import `zodToFieldErrors` / `supabaseErrorKey` / `IssuesActionResult` / `IssuesActionFailure` from `src/features/issues/map-errors.ts` and `src/features/issues/types.ts`. Renaming or moving those files breaks four other features. The shared concerns probably belong in `src/lib/` or `src/features/_shared/`.
2. **`api/test-supabase/route.ts` is a live diagnostic.** It selects from `pg_tables` using the cookie-bound Supabase client and returns the result on `GET /api/test-supabase`. It is reachable from any signed-in user (the proxy excludes `/api`, so it has no auth gate beyond Supabase RLS). It should either be deleted, gated behind a flag, or restricted to local development.
3. **`OPS_EXPERIMENTAL_UI` is read but unused.** `src/lib/env.ts` exposes `isExperimentalUiFlagSet()`, surfaced in the super-admin settings panel (`src/app/[locale]/admin/settings/page.tsx` line 45), but no runtime code branches on it. The flag exists for display only — easy to mistake for a real toggle.
4. **`OPS_DEMO_RESET_ENABLED` defaulting.** Default is "enabled outside production" (`src/lib/env.ts` `isDemoResetEnabled`). In production, an explicit `OPS_DEMO_RESET_ENABLED=true` is required; missing the variable disables reset. The implementation is correct, but it is the only env var with this kind of inverted production default — easy to misread.
5. **`resetDemoData` deletes by trick predicate.** `src/features/settings/actions.ts` lines 48–73 use `delete().neq("id", NIL_UUID)` and `delete().gte("created_at", "1970-01-01T00:00:00Z")` because PostgREST refuses unconditional `DELETE`. The intent is "delete everything"; if the predicate is ever copy-pasted into a guarded delete, it still wipes the table. Treat this file as a special case.
6. **Two issue-detail routes implement the same UI.** `src/app/[locale]/issues/[id]/page.tsx` and `src/app/[locale]/projects/[projectKey]/issues/[issueNumber]/page.tsx` both build a `QueryClient`, prefetch, and render the same `IssueDetailPanel` with the same permission props. Any change in prefetching or permissions has to be made in both files; they have already drifted slightly (one resolves an issue first, the other does not).
7. **List-URL parameters only support offset mode.** `src/features/issues/list-url-params.ts` `issuesListParamsToSearchParams` throws if `mode !== "offset"`. The schema (`listIssuesSchema`) and service support both modes; the URL syncing layer does not. This is a one-way door for any feature that wants cursor mode in the URL.
8. **Cursor pagination is offset in disguise.** `encodeListCursor` in `src/features/issues/service.ts` returns `base64url(JSON({ o: offset }))`. It is forward-compatible with keyset pagination but currently provides no benefit over offset, while *looking* like a real cursor to consumers.
9. **`ISSUES_CACHE_TAG` is wired for writes but never produced.** `src/features/issues/cache.ts` defines the tag; every issue mutation calls `revalidateTag(ISSUES_CACHE_TAG, "max")`. No `fetch(..., { next: { tags: [...] } })` actually emits the tag. Today the tag is a no-op. New `fetch`-based caches must opt in.
10. **`src/features/*/service.ts` is server-only by convention only.** None of the service files have `import "server-only"`. They import `@/lib/supabase/server` (which calls `cookies()` from `next/headers`), so a client import would fail at build time — but the protective marker is missing.
11. **RLS and service duplication for role updates.** `src/features/users/service.ts` `updateUserRole` re-implements the admin-only-edits-`user`-rows rule (lines 66–76). The same rule is in the SQL policy `phase8_user_profiles_update_admin_user_rows`. If one is changed and the other is not, the discrepancy will manifest as either UI errors or silent RLS rejections.
12. **`user_profiles` insert/delete have no RLS policies in migrations.** The Phase 8 migration only writes SELECT and UPDATE policies; signup-side INSERT is expected to come from a trigger or app code outside this repo (`docs/SUPABASE_MIGRATIONS.md` §7 documents the optional `handle_new_user` pattern). If a project enables RLS on `user_profiles` without that trigger, signup will fail to create profile rows and `getUserAuthContext` will return `null` (logging the dev-only `[auth] user_profiles: no row visible` warning in `src/lib/auth/session.ts`).
13. **No request ID, no correlation log line.** Audit rows include `actor_id` and `created_at` only; combining audit with platform logs requires correlating by time and user.
14. **CI does not run tests.** `.github/workflows/ci.yml` runs lint + tsc and explicitly skips Playwright browser downloads. The Playwright spec (`e2e/critical-path.spec.ts`, single file) is meaningful only when run locally with `E2E_EMAIL` / `E2E_PASSWORD` set.
15. **Optimistic updates roll back only the detail cache.** `useUpdateIssue.onError` restores the detail entry but does not touch list caches; `onSuccess` invalidates lists. If an update fails between the optimistic patch and a list re-render, the list briefly shows the old data while the detail is reverted (mostly harmless because the failed update never landed in the list cache).
16. **`src/proxy.ts` matcher excludes any path containing a dot** (`.*\\..*`). This is the standard Next pattern for static assets, but it also means custom routes with literal dots in the segment will silently bypass auth refresh and locale handling.

## 17. Quality gates and tooling

- **TypeScript:** `npm run tsc` (`tsc --noEmit`); strict mode with `noUnusedLocals`, `noUnusedParameters`, `noImplicitReturns`, `noFallthroughCasesInSwitch` (`tsconfig.json`).
- **ESLint:** `npm run lint` (`eslint`); flat config in `eslint.config.mjs` with `simple-import-sort` and `unused-imports`.
- **E2E:** `npm run test:e2e` (Playwright, `e2e/critical-path.spec.ts`). Skips automatically when `E2E_EMAIL`/`E2E_PASSWORD` are unset.
- **No unit tests** are present in the repo at the time of this writing.

The architecture rule (`.cursor/rules/architecture.mdc`) lists the same gates as the definition of "done", plus translations parity and an audit entry per mutation.
